# VoynichLab Repository Inventory

This inventory is the first map a maintainer should read before moving files.
VoynichLab is not a normal single app repo: it contains live annotation tools,
research laboratories, frozen evidence, public artifacts, and a static portal.

For the operating model that explains how these areas may depend on each other,
see `docs/MONOREPO-ARCHITECTURE.md`.

For the protected annotation tool map, see `docs/DATASETCREATOR-ARCHITECTURE.md`.

For ignored local state and disk-usage policy, see `docs/LOCAL-STATE-INVENTORY.md`.

## Top-Level Areas

| Path | Role | Git policy | Cleanup risk |
|---|---|---|---|
| `DataSetCreator/` | Tauri/Rust annotation tool and local labeling workflow | Source code can be committed intentionally; local DBs, page assets, backups, and generated annotation assets must not be moved by cleanup work | Very high |
| `EVAComparisonLab/` | ATOMS/EVA extraction, comparison, snapshots, current lab outputs | Source scripts and frozen reports are commit-worthy; `current` outputs need review before publishing | High |
| `GrammarDiscoveryLab/` | Grammar discovery, frozen GRAMMAR-V1 material, validation scripts | Commit reproducible scripts and frozen scientific milestones | Medium |
| `TranslationLab/` | Experimental translation-facing tooling | Keep outputs/cache local unless explicitly promoted | Medium |
| `research/` | Audits, corpus revisions, frozen research evidence | Frozen releases are immutable; working audit folders are scratch until promoted | High |
| `research/audits/` | Canonical cross-lab anomaly and quality-control ledgers | Commit reviewed decisions; never maintain duplicate active ledgers elsewhere | High |
| `research-feed/` | Public experiment, milestone, release registry | Commit only when publishing or correcting public metadata | Medium |
| `artifacts/public/` | Public reproducibility bundles consumed by portal | Generated by `research:build`; commit only as part of named experiment/release | Medium |
| `apps/portal/` | Static public portal deployed at Vercel | Commit source and intentional public data; never commit `.vercel/` | Medium |
| `apps/portal/data/` | Static data generated from registry/artifacts | Generated; commit only when needed for public portal deployment | Medium |
| `packages/lab-exporter/` | Registry/artifact/portal exporter and repo guardrails | Commit like normal source code | Low |
| `paper/` | Preprint planning and manuscript material | Commit intentional scientific writing | Low |
| `.github/` | GitHub workflows and automation | Commit intentionally; avoid workflows that mutate evidence automatically | Medium |

## Frozen Evidence Discovery

The canonical inventory of corpus snapshots, corpus releases, model freezes,
and validation releases is `research/FROZEN-EVIDENCE.md`. Its machine-readable
companion is `research/frozen-evidence.json`.

Older freezes retain historical lab-local paths for provenance. New corpus
freezes beginning with Corpus V3 must be created under `research/frozen/`.

## Protected Assets

Do not delete, move, rebase, rewrite, or auto-format these without explicit user
approval and a restoration plan:

- any `*.db`, `*.sqlite`, or `*.sqlite3` file;
- `DataSetCreator/manuscript-pages-yale/`;
- `DataSetCreator/backups/`;
- local DatasetCreator evidence/export directories;
- `research/frozen/**`;
- already published tags and frozen release artifacts;
- any file containing manual correction ledgers.

## Working Buckets

Every cleanup pass should classify files into one of these buckets before any
commit:

1. Source code.
2. Public registry metadata.
3. Public generated artifacts.
4. Portal generated data.
5. Frozen research evidence.
6. Local scratch/current lab output.
7. High-risk annotation data.

If a diff contains bucket 1 and thousands of files from bucket 6, split the work.
